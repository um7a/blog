Overview

Kubernetes workloads are resources used to run and manage applications within a Kubernetes cluster. This article introduces these resources, including Pods, ReplicaSets, Deployments, StatefulSets, DaemonSets, Jobs, and CronJobs.

Pods

A Pod is the smallest unit of workload resources.

A Pod consists of one or more containers. In general, it is recommended to create a Pod with one container. However, if you want to run containers that support your main container, you can add them to the same Pod. This pattern is called the sidecar pattern, ambassador pattern, or adapter pattern, depending on the role of the supporting containers.

Containers in the same Pod share the same network namespace, so they can communicate with each other by using localhost.

Create a Pod

To create a Pod, you need a manifest file. Create a file named pod.yaml with the following contents:

---
apiVersion: v1
kind: Pod
metadata:
  name: sample-pod
spec:
  containers:
    - name: nginx-container
      image: nginx:1.30

Apply the manifest to a Kubernetes cluster:

kubectl apply -f pod.yaml

The message pod/sample-pod created should be displayed.

Get Pod Information

kubectl get pods -o wide

Get Pod Logs

The following command displays the log messages of a Pod. Kubernetes collects log messages written to the container’s stdout and stderr streams.

kubectl logs sample-pod -f

Run Commands in Container

Commands can be executed inside the container using kubectl exec:

kubectl exec sample-pod -- ls /var/log/nginx/access.log

Like the docker exec command, kubectl exec supports the -i option and -t options.
The -i option keeps stdin open and passes standard input to the process running inside the container. The -t option allocates a TTY, enabling interactive terminal sessions.

kubectl exec -it sample-pod -- /bin/bash

Specify ENTRYPOINT and CMD of the Container

command is used to specify the container’s ENTRYPOINT, while args is used to specify the container’s CMD:

---
apiVersion: v1
kind: Pod
metadata:
  name: sample-pod-command-and-args
spec:
  containers:
    - name: nginx-container
      image: nginx:1.30
      command: ["/bin/sleep"]  # ENTRYPOINT
      args: ["10"]             # CMD

Specify Working Directory of the Container

workingDir is used to specify the container's WORKDIR:

---
apiVersion: v1
kind: Pod
metadata:
  name: sample-pod-workingdir
spec:
  containers:
    - name: nginx-container
      image: nginx:1.30
      workingDir: /tmp        # WORKINGDIR

The following command confirms that the working directory is set to /tmp:

kubectl exec -it sample-pod-workingdir -- pwd

/tmp

Create a Pod using Host Network

The IP address assigned to a Pod is in a different range from the host's IP address. It is also unreachable from outside the Kubernetes cluster. To allow a Pod to use the host's network, set spec.hostNetwork: true in the Pod manifest:

---
apiVersion: v1
kind: Pod
metadata:
  name: sample-pod-host-network
spec:
  hostNetwork: true    # Here
  containers:
    - name: nginx-container
      image: nginx:1.30

Apply the manifest:

kubectl apply -f pod-host-network.yaml

Then confirm that an HTTP request can reach the Nginx container:

curl http://<KUBERNETES_NODE_IP_ADDRESS>/ -v

ReplicaSets

Next, let’s look at ReplicaSets.

A ReplicaSet is a workload resource that creates Pod replicas and keeps the number of running Pods at the user-specified count.

Kubernetes attempts to distribute the Pods managed by a ReplicaSet evenly across the available nodes.

Create a ReplicaSet

The manifest for a ReplicaSet looks like this:

---
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: sample-replicaset
spec:
  replicas: 3  # The number of pods.
  selector:
    matchLabels:
      app: sample-app  # Label that Pods in this ReplicaSet must have.
  template:
    metadata:
      labels:
        app: sample-app  # This line sets the 'sample-app' label on the Pods.
    spec:
      containers:
      - name: nginx-container
        image: nginx:1.30

spec.replicas specifies the number of Pods to maintain. spec.selector.matchLabels.app specifies the label that Pods managed by the ReplicaSet must have.

Apply the manifest to create a ReplicaSet:

kubectl apply -f replicaset.yaml

kubectl get replicasets -o wide

One of the benefits of using a ReplicaSet is its self-healing capability. Self-healing is a feature that allows Kubernetes to detect when a Pod managed by a ReplicaSet stops running and automatically create a replacement Pod.

kubectl delete pod sample-replicaset-gdrnx

kubectl get pods -l app=sample-app -o wide

Deployments

A Deployment is a workload resource that manages ReplicaSets. It enables rolling updates and rollbacks.

Create a Deployment

The following snippet shows the manifest for a Deployment:

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
    spec:
      containers:
      - name: nginx-container
        image: nginx:1.30

Apply the manifest to create a Deployment:

kubectl apply -f deployment.yaml

Performing a Rolling Update with Deployments

A rolling update of the Deployment is triggered whenever the Pod template spec.template changes.

To demonstrate a rolling update, update the container image version in the Deployment manifest:

# diff sample-deployment.yaml sample-deployment-version-up.yaml
18c18
<         image: nginx:1.30
---
>         image: nginx:1.31

Apply the version update:

kubectl apply -f sample-deployment-version-up.yaml

The rolling update is now in progress.

The following command shows the progress of the rolling update:

kubectl rollout status deployment sample-deployment

The command produces output similar to the following:

Delete a Deployment

Delete a Deployment using the following command:

kubectl delete deployment sample-deployment

DaemonSets

A DaemonSet is a workload resource that ensures one Pod runs on each eligible node.

A DaemonSet is useful for running monitoring agents or other system-level processes that need to run on every Kubernetes node.

Create a DaemonSet

The following manifest defines a DaemonSet:

---
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: sample-daemonset
spec:
  selector:
    matchLabels:
      app: sample-daemonset
  template:
    metadata:
      labels:
        app: sample-daemonset
    spec:
      containers:
      - name: nginx-container
        image: nginx:1.30

Apply the manifest to create a DaemonSet:

kubectl apply -f daemonset.yaml

Get DaemonSet Information

The following command displays the status of the created DaemonSet:

kubectl get daemonset -o wide

The command produces output similar to the following:

NAME               DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE     CONTAINERS        IMAGES       SELECTOR
sample-daemonset   2         2         2       2            2           <none>          2m41s   nginx-container   nginx:1.30   app=sample-daemonset

The following command verifies that the Pods are distributed across the Kubernetes nodes:

The output shows that the DaemonSet has scheduled one Pod on each node, as indicated by the NODE column.

Delete a DaemonSet

StatefulSets

A StatefulSet is a workload resource that is designed for applications that need to preserve their identity and data across restarts.

Create a StatefulSet

The following manifest defines a StatefulSet with three replicas, each having its own persistent volume:

---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sample-statefulset
spec:
  serviceName: sample-statefulset
  replicas: 3
  selector:
    matchLabels:
      app: sample-statefulset
  template:
    metadata:
      labels:
        app: sample-statefulset
    spec:
      containers:
      - name: nginx-container
        image: nginx:1.30
        volumeMounts:
        - name: nginx-storage
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
  - metadata:
      name: nginx-storage
    spec:
      accessModes:
      - ReadWriteOnce
      resources:
        requests:
          storage: 1G

Apply the manifest to a Kubernetes cluster:

kubectl apply -f statefulset.yaml

Get StatefulSet Information

The following command displays the status of the created StatefulSet:

kubectl get statefulset -o wide

The command produces output similar to the following:

NAME                 READY   AGE     CONTAINERS        IMAGES
sample-statefulset   3/3     9m19s   nginx-container   nginx:1.30

The following command verifies that the Pods are also created:

kubectl get pods -o wide

The command produces output similar to the following:

NAME                   READY   STATUS    RESTARTS   AGE   IP             NODE             NOMINATED NODE   READINESS GATES
sample-statefulset-0   1/1     Running   0          10m   10.244.1.123   worker01.local   <none>           <none>
sample-statefulset-1   1/1     Running   0          10m   10.244.2.16    worker02.local   <none>           <none>
sample-statefulset-2   1/1     Running   0          10m   10.244.2.18    worker02.local   <none>           <none>

One notable characteristic of StatefulSets is that Pod names end with ordinal numbers such as -0, -1, and -2. Kubernetes creates Pods managed by a StatefulSet in ordinal order. While Pod names in a Deployment change when they are recreated, Pod names in a StatefulSet remain the same.

Check PersistentVolumeClaims Used by StatefulSet Pods

The following command displays the status of the PersistentVolumeClaims (PVC) used by the StatefulSet Pods:

kubectl get pvc -o wide

The command produces output similar to the following:

NAME                                 STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE   VOLUMEMODE
nginx-storage-sample-statefulset-0   Bound    pvc-5ed1b9d7-0eb2-4e87-9617-1ab37d766412   1G         RWO            local-path     <unset>                 20m   Filesystem
nginx-storage-sample-statefulset-1   Bound    pvc-f6554d63-89f7-427c-a88c-f799ec9bddef   1G         RWO            local-path     <unset>                 20m   Filesystem

The volumeClaimTemplates section allows Kubernetes to automatically create a dedicated PVC for each Pod in a StatefulSet. If a Pod is deleted, its corresponding PVC remains, allowing the recreated Pod to continue using the same data. Deployments do not provide this capability.

Delete a StatefulSet

The following command deletes a StatefulSet:

kubectl delete statefulset sample-statefulset

Jobs

A Job is a workload resource that creates one or more Pods and ensures that a specified number of them complete successfully.

Create a Job

The following manifest defines a Job:

---
apiVersion: batch/v1
kind: Job
metadata:
  name: sample-job
spec:
  completions: 3  # Number of successful completions required
                  # for the Job to be considered complete.
  parallelism: 1  # Maximum number of Pods that can run
                  # in parallel.
  backoffLimit: 10  # Number of retries before the Job is marked as failed.
  template:
    spec:
      containers:
      - name: busybox-container
        image: busybox:1.37
        command:
        - sh
        - -c
        - |
          echo "Job started"
          sleep 10
          echo "Job completed"
      restartPolicy: Never

Apply the manifest to a Kubernetes cluster:

kubectl apply -f job.yaml

Get Job Information

The following command displays the status of the created Job:

kubectl get job sample-job

The command produces output similar to the following:

NAME         STATUS     COMPLETIONS   DURATION   AGE
sample-job   Complete   3/3           42s        2m26s

Check Pods Managed by Job

The following command continuously monitors the Pods managed by the Job:

kubectl get pods --watch

CronJobs

A CronJob is a workload resource that creates Jobs on a repeating schedule.

Create a CronJob

The following manifest defines a CronJob:

---
apiVersion: batch/v1
kind: CronJob
metadata:
  name: sample-cronjob
spec:
  schedule: "*/1 * * * *"  # Run every minute
  concurrencyPolicy: Allow  # Concurrency policy for the CronJob. Valid values are:
                            # - Allow: Allows Jobs to run concurrently. This is the default value.
                            # - Forbid: Skips the next run if the previous Job is still running.
                            # - Replace: Replaces the currently running Job with a new one.
  startingDeadlineSeconds: 30  # Deadline in seconds for starting a Job
                               # if it misses its scheduled time.
  successfulJobsHistoryLimit: 5  # Number of successful Jobs to retain.
  failedJobsHistoryLimit: 3  # Number of failed Jobs to retain.
  suspend: false
  jobTemplate:
    spec:
      completions: 1
      parallelism: 1
      backoffLimit: 0
      template:
        spec:
          containers:
          - name: busybox-container
            image: busybox:1.37
            command:
            - sh
            - -c
            - |
              echo "Job started"
              sleep 10
              echo "Job completed"
          restartPolicy: Never

Apply the manifest to a Kubernetes cluster:

kubectl apply -f cronjob.yaml

Get CronJob Information

The following command displays the status of the created CronJob:

kubectl get cronjob sample-cronjob

The command produces output similar to the following:

NAME             SCHEDULE      TIMEZONE   SUSPEND   ACTIVE   LAST SCHEDULE   AGE
sample-cronjob   */1 * * * *   <none>     False     1        3s              73s

The following command continuously monitors the Jobs managed by the CronJob:

kubectl get job

The command produces output similar to the following:

NAME                      STATUS     COMPLETIONS   DURATION   AGE
sample-cronjob-29711434   Complete   1/1           13s        2m8s
sample-cronjob-29711435   Complete   1/1           14s        68s
sample-cronjob-29711436   Running    0/1           8s         8s

Suspend a CronJob

To suspend a CronJob, set spec.suspend to true:

spec:
  suspend: true

Apply the updated manifest:

kubectl apply -f cronjob.yaml

Check the status of the CronJob:

kubectl get cronjob sample-cronjob

The CronJob becomes suspended, as indicated by True in the SUSPEND column:

NAME             SCHEDULE      TIMEZONE   SUSPEND   ACTIVE   LAST SCHEDULE   AGE
sample-cronjob   */1 * * * *   <none>     True      0        41s             8m51s

Manually Execute a CronJob

kubectl create job with the --from option creates a Job from a CronJob:

kubectl create job sample-job-from-cronjob --from cronjob/sample-cronjob

Delete a CronJob

The following command deletes a CronJob:

kubectl delete cronjob sample-cronjob
