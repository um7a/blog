Overview

In this post, I will walk you through how to set up a Kubernetes worker node on Rocky Linux 10 with a Raspberry Pi 4.

Here's the overall flow of the setup:

  • Configure the Linux system required for a Kubernetes worker node
  • Install the container runtime (containerd)
  • Install Kubernetes components (kubeadm, kubelet, and kubectl)
  • Join the worker node to the Kubernetes cluster

Let's dive into the setup.

1. Configuration

  • Hardware : Raspberry Pi 4 (8GB RAM)
  • OS : Rocky Linux 10

2. Linux Settings for a Kubernetes Worker Node

2.1. Disable Swap

The default behavior of kubelet is to fail to start if swap memory is detected on a node. This means that swap should either be disabled or explicitly tolerated by kubelet. I chose the former.

However, things are a bit different on Rocky Linux running on a Raspberry Pi.

In a typical environment, swap is controlled by /etc/fstab, so you can disable it by editing this file. However, on Rocky Linux running on a Raspberry Pi, swap is configured as a disk partition and is not managed via /etc/fstab.

To disable swap in this environment, you can mask swap.target using the following command:

# Not allow systemd to use swap after reboot
$ sudo systemctl mask swap.target

Reboot the system and verify that swap is disabled:

$ sudo reboot
$ swapon --show
# -> should be empty

2.2. Set SELinux to permissive

Setting SELinux to permissive mode is required to allow containers to access the host filesystem; for example, some cluster network plugins require this. You have to do this until SELinux support is improved in the kubelet. Or you can leave SELinux enabled if you know how to configure it but it may require settings that are not supported by kubeadm.

Set SELinux to permissive mode by running the following command:

# Set SELinux to permissive after reboot
$ sudo sed -i 's/^SELINUX=enforcing$/SELINUX=permissive/' /etc/selinux/config

Reboot the system and verify that SELinux is in permissive mode:

$ sudo reboot
$ getenforce
# -> Permissive

2.3. Enable IPv4 Packet Forwarding

By default, the Linux kernel does not allow IPv4 packets to be routed between interfaces. Most Kubernetes cluster networking implementations will change this setting, but some might expect the administrator to do it manually.

Set the sysctl parameter to allow IPv4 packets to be routed between interfaces:

$ cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.ipv4.ip_forward = 1
EOF

Apply sysctl params without reboot:

$ sudo sysctl --system

2.4. Enable memory cgroup controller

On Rocky Linux 10 running on a Raspberry Pi, cgroup_disable=memory is set in the kernel command line parameters, and this causes an error during kubeadm initialization.

$ cat /proc/cmdline
reboot=w coherent_pool=1M 8250.nr_uarts=1 pci=pcie_bus_safe cgroup_disable=memory numa_policy=interleave bcm2708_fb.fbwidth=640 bcm2708_fb.fbheight=480 bcm2708_fb.fbdepth=16 bcm2708_fb.fbswap=1 smsc95xx.macaddr=2C:CF:67:DC:D6:11 vc_mem.mem_base=0x3fc00000 vc_mem.mem_size=0x40000000  console=ttyAMA0,115200 console=tty1 root=LABEL=RPIROOT rootfstype=ext4 rootwait

To enable the memory cgroup controller, add cgroup_enable=memory cgroup_memory=1 at the end of /boot/efi/cmdline.txt.

# Add `cgroup_enable=memory cgroup_memory=1` at the end
$ sudo vi /boot/efi/cmdline.txt
console=ttyAMA0,115200 console=tty1 root=LABEL=RPIROOT rootfstype=ext4 rootwait cgroup_enable=memory cgroup_memory=1

Confirm the kernel command line parameters are updated:

$ sudo reboot
$ cat /proc/cmdline
reboot=w coherent_pool=1M 8250.nr_uarts=1 pci=pcie_bus_safe cgroup_disable=memory numa_policy=interleave bcm2708_fb.fbwidth=640 bcm2708_fb.fbheight=480 bcm2708_fb.fbdepth=16 bcm2708_fb.fbswap=1 smsc95xx.macaddr=2C:CF:67:DC:D6:11 vc_mem.mem_base=0x3fc00000 vc_mem.mem_size=0x40000000  console=ttyAMA0,115200 console=tty1 root=LABEL=RPIROOT rootfstype=ext4 rootwait cgroup_enable=memory cgroup_memory=1

2.5. Load kernel module for Flannel

The br_netfilter kernel module is required by Flannel. Load this kernel module with the following command:

$ echo br_netfilter | sudo tee /etc/modules-load.d/br_netfilter.conf

Reboot the system and verify that the br_netfilter module is loaded:

$ sudo reboot
$ sudo lsmod | grep br_netfilter

2.6. Set sysctl params for Flannel

Set the sysctl parameters required by Flannel with the following commands:

$ cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.ipv4.ip_forward = 1
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
EOF

Apply sysctl params without reboot:

$ sudo sysctl --system

2.7. Allow Port 10250 from the Control Plane

Port 10250 on the worker node is used by the control plane node to access the kubelet API. Allow access to this port using the following commands:

$ sudo firewall-cmd --permanent --add-port=10250/tcp
$ sudo firewall-cmd --reload

These are the required Linux settings. Let's move on to the Kubernetes components installation.

3. Install containerd

Add the docker-ce repository.

$ sudo dnf install -y yum-utils
$ sudo dnf config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo

Install containerd from the docker-ce repository.

$ sudo dnf install -y containerd.io

Generate the default containerd configuration.

$ sudo containerd config default | sudo tee /etc/containerd/config.toml

Enable and start containerd.

4. Install Kubernetes components (kubeadm, kubelet, and kubectl)

Add Kubernetes repository.

$ cat <<EOF | sudo tee /etc/yum.repos.d/kubernetes.repo
[kubernetes]
name=Kubernetes
baseurl=https://pkgs.k8s.io/core:/stable:/v1.35/rpm/
enabled=1
gpgcheck=1
gpgkey=https://pkgs.k8s.io/core:/stable:/v1.35/rpm/repodata/repomd.xml.key
exclude=kubelet kubeadm kubectl cri-tools kubernetes-cni
EOF

The exclude parameter in the repository definition ensures that Kubernetes-related packages are not upgraded automatically when running dnf update, because Kubernetes upgrades should follow a specific procedure.

Install kubelet, kubeadm, and kubectl.

$ sudo dnf install -y \
kubelet \
kubeadm \
kubectl \
--disableexcludes=kubernetes

Start and enable kubelet before running kubeadm.

$ sudo systemctl enable --now kubelet

5. Add the Linux Worker Node to the Kubernetes Cluster

Run the following command on the control plane node to retrieve the kubeadm join command:

$ kubeadm token create --print-join-command
# -> This will output something like:
#    kubeadm join 192.168.1.2:6443 --token  --discovery-token-ca-cert-hash sha256:

Run this command on the worker node (with sudo):

$ sudo kubeadm join 192.168.1.2:6443 --token  --discovery-token-ca-cert-hash sha256:

A few seconds later, you should see the worker node in the output of kubectl get nodes. Run kubectl on the control plane node.

6. Launch the First Pod

Now you can launch the first pod on the worker node. Run the following command on the control plane node:

$ kubectl run test-pod --image=busybox --restart=Never -- sleep 3600

You can then verify that the first pod is running on the worker node with the following command:

$ kubectl get pods -o wide
NAME       READY   STATUS    RESTARTS   AGE   IP           NODE             NOMINATED NODE   READINESS GATES
test-pod   1/1     Running   0          15s   10.244.1.2   worker01.local

You can check the pod logs; however, since it only executes the sleep 3600 command, the log is empty.

$ kubectl logs test-pod
# -> Should be empty.
