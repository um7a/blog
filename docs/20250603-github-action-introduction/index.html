Introduction

In this article, I'd like to delve into GitHub Actions.

Components of GitHub Actions

The followings are the components of GitHub Actions.

  • Workflow
    Workflows enable you to automate various tasks such as building, testing, and deploying code. A workflow is defined using a YAML file in your repository's .github/workflows directory.

  • Event
    Events trigger workflows. You can specify event names in a workflow's YAML file to trigger the workflow with the events. Events can be grouped into the following categories:

    • Code-related events
      These events are related with code changes.
      These events include push, pull_request, pull_request_review.

    • Issue and project management events
      These events are related to issue tracking and project management.
      These events include issues, issue_comment, milestone.

    • Scheduled events
      These events are triggered based on a schedule. They alow you to run workflows at specific intervals.
      These events include schedule.

    • Manual events
      These events are triggered manually. They allow you to run workflows on-demand.
      These events include workflow_dispatch.

    • Repository and organization management events
      These events are related to repository or organization management.
      These events include repository, team, organization.

    • External events
      These events are triggered from external services.
      These events include repository_dispatch, workflow_run.

  • Job
    A job represents a set of related steps you want to execute in a workflow.

  • Step
    Steps are the sequence of tasks, such as checking out code and running a script.

Let's delve into each component.

Workflows

Let's create a new repository and add .github/workflows/nodejs.yml to it.

.github
└── workflows
    └── nodejs.yml

The following is an example workflow for building and testing a Node.js application.

name: Node.js CI
run-name: ${{ github.actor }} is running Node.js CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20.x'
      - run: npm ci
      - run: npm run build --if-present
      - run: npm test

Properties of workflows

The following properties are available for configuring workflows:

  • name
    The name of the workflow as it will appear in the "Actions" tab of the GitHub repository. If this field is omitted, the name of the workflow file will be used instead.

  • run-name
    The name for workflow runs generated from the workflow, which will appear in the list of workflow runs on your repository's "Actions" tab.

  • on
    Specifies the trigger for this workflow. The above example uses the push event, so a workflow run is triggered every time someone pushes a change to the repository or merges a pull request.

    You can specify options for events. For example, the following configuration specifies branch names for the push event that trigger the workflow:

    on:
  push:
    branches:
      - main
      - releases/**

    You can also specify multiple events. With the following setting, workflow run is triggered by push and pull_request event.

    on:
  push:
    branches:
      - main
      - releases/**
  pull_request:
    types:
      - opened
      - reopened

  • jobs
    Specifies all the jobs that run in the workflow.

Jobs

Jobs are defined in jobs property of workflow YAML file.

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20.x'
      - run: npm ci
      - run: npm run build --if-present
      - run: npm test

Properties of jobs

The properties of jobs include the following:

  • Job name
    In the above configuration, build is the name of the job. You can specify arbitrary name for the job.

  • runs-on
    Specifies the name of the runner on which the job will be executed. You can specify a GitHub-hosted runner, or a self-hosted runner.

  • steps
    Specifies all the steps that run in the job.

  • needs
    Jobs within a workflow run in parallel by default. If you want to define dependencies between jobs, you can use needs property with a job name.

    jobs:
  build:
    runs-on: ubuntu-lates
    steps:
      ...
  test:
    needs: build
    runs-on: ubuntu-lates
    steps:
      ...

Steps

Steps are defined in steps property as the member of jobs property.

jobs:
  build:
    ...
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20.x'
      - run: npm ci
      - run: npm run build --if-present
      - run: npm test

Properties of steps

The properties of steps include the following:

  • run
    Specify the command or script to execute.

  • uses
    Specify the identifier of the action to use. Built-in actions are provided by GitHub. Community and third-party actions are also available. It's identifier is the format of <user>/<repo>@<version>.

  • with
    Specify a set of parameters to pass to the action.

  • if
    Specify a conditional expression that determines whether the step should be executed.

Conclusion
