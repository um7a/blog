Overview

I’m continuing my Kubernetes learning journey, and recently I wanted to experiment with StatefulSets. Since StatefulSets typically require persistent storage, I needed a simple way to provision PersistentVolumes (PVs) in my Kubernetes cluster. In this article, I’ll walk through installing and testing rancher/local-path-provisioner.

What is rancher/local-path-provisioner?

local-path-provisioner is a lightweight dynamic provisioner that provisions PVs from local directories on Kubernetes nodes.

Install local-path-provisioner

Apply the official manifest:

kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml

The output should look similar to the following:

namespace/local-path-storage created
serviceaccount/local-path-provisioner-service-account created
role.rbac.authorization.k8s.io/local-path-provisioner-role created
clusterrole.rbac.authorization.k8s.io/local-path-provisioner-role created
rolebinding.rbac.authorization.k8s.io/local-path-provisioner-bind created
clusterrolebinding.rbac.authorization.k8s.io/local-path-provisioner-bind created
deployment.apps/local-path-provisioner created
storageclass.storage.k8s.io/local-path created
configmap/local-path-config created

The manifest creates the following resources:

  • Namespace: local-path-storage
  • Deployment: local-path-provisioner
  • StorageClass: local-path

The Deployment is the main component of the local-path-provisioner system. It watches for new PersistentVolumeClaims (PVCs) and automatically provisions PVs for them. When a Pod that uses a PVC is scheduled to a node, the provisioner creates a corresponding PV and a directory on that node. By default, the directory is created under /opt/local-path-provisioner/pvc-<uuid>/. When the PVC is deleted, the associated directory may also be removed, depending on the reclaim policy.

Verify the Deployment:

kubectl get deployments -n local-path-storage

The output should look similar to the following:

NAME                     READY   UP-TO-DATE   AVAILABLE   AGE
local-path-provisioner   1/1     1            1           4d5h

Check the StorageClass:

kubectl get storageclass

The output should look similar to the following:

NAME         PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-path   rancher.io/local-path   Delete          WaitForFirstConsumer   false                  4d5h

Create a test PVC

Create a manifest with the following content:

---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-pvc
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: local-path
  resources:
    requests:
      storage: 1Gi

Apply it:

kubectl apply -f test-pvc.yaml

Verify the PVC:

kubectl get pvc

You can see output similar to the following:

NAME       STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE
test-pvc   Pending                                      local-path     <unset>                 3m54s

At this point, the PVC is in the Pending state because the local-path StorageClass uses the WaitForFirstConsumer volume binding mode. Since local-path storage is tied to a specific node, Kubernetes delays volume binding until a Pod that uses the PVC is scheduled.

Use the PVC from a Pod

Let's test the PVC by creating a Pod that mounts it.

Create a manifest for a Pod that uses the test PVC:

---
apiVersion: v1
kind: Pod
metadata:
  name: storage-test-pod
spec:
  containers:
  - name: busybox
    image: busybox
    command:
    - sleep
    - "3600"
    volumeMounts:
    - name: data
      mountPath: /data

  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: test-pvc

Apply:

kubectl apply -f storage-test-pod.yaml

Verify the PVC is in the Bound state:

kubectl get pvc

Output should look similar to the following:

NAME       STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS   VOLUMEATTRIBUTESCLASS   AGE
test-pvc   Bound    pvc-50612313-fafd-4dc6-8256-65388fcab05b   1Gi        RWO            local-path     <unset>                 11m

Next, verify that Kubernetes has created a PV:

kubectl get pv

Output should look similar to the following:

NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM              STORAGECLASS   VOLUMEATTRIBUTESCLASS   REASON   AGE
pvc-50612313-fafd-4dc6-8256-65388fcab05b   1Gi        RWO            Delete           Bound    default/test-pvc   local-path     <unset>                          3m3s

Create a file in the volume:

kubectl exec -it storage-test-pod -- sh
echo hello > /data/test.txt

Delete the Pod and then recreate it:

kubectl delete pod storage-test-pod
kubectl apply -f storage-test-pod.yaml

The file should still exist because it is stored on the PV rather than inside the Pod:

kubectl exec -it storage-test-pod -- sh
cat /data/test.txt

The output should look similar to the following:

hello

Set local-path as the Default Storage Class

The following command sets local-path as the default storage class:

kubectl annotate storageclass local-path \
  storageclass.kubernetes.io/is-default-class=true

The output shows that local-path is the default StorageClass, as indicated by (default) in the NAME column:

NAME                   PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-path (default)   rancher.io/local-path   Delete          WaitForFirstConsumer   false                  11d

Limitations of local-path-provisioner

local-path-provisioner stores data on the node where a Pod is running. If that node becomes unavailable, the Pod can no longer access its storage because the data resides on that node's local disk.

For this reason, local-path-provisioner is an excellent choice for:

  • Learning Kubernetes
  • Home labs
  • Single-node clusters

However, it is not ideal for highly available storage solutions.

Common next steps for implementing highly available storage include:

  • NFS
  • Longhorn
  • Rook + Ceph
